# Repository Guidelines

## Project Structure & Module Organization
This repository is a Flux-managed homelab GitOps repo.

- `clusters/homelab/`: Flux entrypoints and cluster-level reconciliation. `flux-system/gotk-sync.yaml` is the root source definition.
- `infrastructure/`: shared platform components such as Cilium, Traefik, Longhorn, CNPG, VolSync, Loki, and cert-manager.
- `apps/`: user-facing workloads and app-specific manifests such as Forgejo, Authelia, Vaultwarden, Homepage, Gatus, and Paperless.
- `docs/`: operational and recovery documentation.
- Encrypted secrets use `*.enc.yaml`; cluster bootstrap inputs include `talconfig.yaml`, `talsecret.enc.yaml`, and local kube/Talos configs.

## Build, Test, and Development Commands
There is no single app build step; validate manifests before pushing.

- `kubectl kustomize infrastructure` checks rendered infrastructure manifests.
- `kubectl kustomize apps` checks rendered app manifests.
- `kubectl diff -k infrastructure` previews live infrastructure changes.
- `kubectl diff -k apps` previews live app changes.
- `kubectl get gitrepository,kustomizations,helmreleases -A` verifies Flux health.
- `kubectl get pods -A` is the quickest runtime health check after reconciliation.

Prefer changing files in the repo, then letting Flux apply them. Avoid treating live `kubectl patch` as the source of truth.

## Coding Style & Naming Conventions
- Use 2-space YAML indentation and keep manifests ordered consistently.
- Keep one app or component per directory with `kustomization.yaml` listing all resources explicitly.
- Name commits with a lowercase type prefix, e.g. `fix: ...`, `docs: ...`.
- Preserve generated Flux comments such as `# This manifest was generated by flux. DO NOT EDIT.`

## Testing Guidelines
- Run `kubectl kustomize` before every commit to catch broken YAML or Kustomize references.
- Use `kubectl diff -k ...` for change review when cluster access is available.
- After merge or push, confirm Flux applied the expected revision and that pods are healthy.
- For policy or storage changes, verify both GitOps status and runtime behavior.

## Commit & Pull Request Guidelines
- Keep commits focused and operationally clear: `fix: allow paperless and kube-state-metrics traffic`.
- In pull requests, include:
  - the affected paths
  - expected cluster impact
  - rollback notes for risky storage, network, or Flux changes
  - screenshots only for UI-facing app changes

## Security & Configuration Tips
- Do not commit plaintext secrets; use SOPS-encrypted `*.enc.yaml`.
- Keep Flux pointing at GitHub HTTPS in `clusters/homelab/flux-system/gotk-sync.yaml`.
- Be careful with recovery docs and bootstrap commands: outdated Flux source commands can reintroduce drift or rollback loops.
